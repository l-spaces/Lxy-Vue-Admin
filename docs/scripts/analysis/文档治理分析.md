# 文档治理分析

## 简介

本文档记录 `docs/scripts` 在本次整理中的一致性分析结论，目标是让文档与源码保持同步且可维护。

## 适用范围

- 文档治理
- 文档与源码一致性复核
- 新增文档前的结构判断

## 对应源码目录或关键文件

- `apps/web-antd/src`
- `packages/*`
- `internal/*`
- `docs/scripts/*`
- `docs/.vitepress/config.mts`

## 核心机制或功能说明

### 盘点结论（本轮）

- `guide`：已按启动、路由、状态、HTTP、i18n、偏好配置重写
- `api`：从单页大文档拆分为请求封装/模块组织/核心接口
- `components`：按包能力与应用层组件分离，避免把工具包写成组件
- `analysis`：保留结构化分析，去除模板化泛文
- `development/deployment/maintenance/reference`：改为可执行目录化文档

### 一致性风险点

- 根导出 `#/api` 仅包含 `core`，历史文档常误写为“导出全部 API”
- `system/dict` 当前没有 `index.ts`，只能按文件导入
- 组件文档中旧的演示占位写法（如 `DemoPreview`）与当前站点内容无直接对应

### 文档治理原则

- 以导出入口和实现代码为准，不以历史描述为准
- 同类信息单点维护，避免多文档互相矛盾
- 文档新增后必须同步 `vitepress` 配置与 `reference/文档目录.md`

## 使用方式、扩展方式或注意事项

- 发现文档不一致时，先定位对应源码文件，再修正文档，不做“经验补写”。
- 若结构变化较大，优先改 `reference` 与导航入口，再分批改正文。
- 对尚无稳定导出的能力，文档应明确标注“以实际实现为准”。
